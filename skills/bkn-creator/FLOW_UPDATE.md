# 更新流程（Update）

## 流程目标

在不破坏现有网络完整性的前提下，执行可追踪、可回滚策略化的变更。

## 验证阶段：流程路由确认（进入更新流程前）

- MUST：先回显“识别结果 + 路由目标 + 输入摘要”，并发起确认请求
- MUST：仅当用户明确确认“进入更新流程”后，才可进入后续阶段
- Never：在验证阶段未确认前输出 `discover/preview/confirm` 细化内容或执行任何更新动作

## 渐进式阶段

1. `discover`：定位 `kn_id` 与待更新范围（网络 / 对象类 / 关系类 / 动作类）
2. `preview`：展示变更前后差异（diff）与影响面
3. `confirm`：等待用户明确“确认更新”
4. `execute`：委托子技能执行更新
5. `verify`：校验更新生效 + 引用完整性
6. `report`：输出结果、失败项、补救建议

> 统一门禁规则见 `./COMMON_RULES.md`。对用户可见输出必须使用“用户回显模板（统一格式）”。

进入条件（前置）：验证阶段路由确认已通过。

## 展示策略（MUST）

- 默认仅输出摘要回显（变更范围、影响面、核验结论、下一步）
- 同一轮仅允许一种主展示格式，禁止同一数据同时用表格与 YAML/JSON 重复展示
- 仅当用户明确请求时才输出详情（如“展开详情”“导出 YAML”“给我表格”）
- 详情输出一次仅允许一种格式（表格 或 YAML/JSON）

## 执行策略

- 小范围属性修改：优先对象/关系级更新
- 结构性变更（新增或重构对象关系）：走“草案重生成 + 校验 + 推送”
- 更新前 MUST 展示影响面：
  - 受影响对象类 / 关系类列表
  - 受影响字段（含 mapped field）
  - 可能触发的完整性风险

## 核验清单（MUST）

1. 目标项存在且 ID 完整
2. 关系引用对象仍然存在
3. 映射字段在目标视图可解析
4. comment/Description 未被意外清空
5. 必要时通过 `kweaver bkn validate`

## 失败处理

- 若更新部分成功，必须分项报告成功/失败并给出重试方案
- Never 在未确认前执行写操作
- Never 自动更新网络；任何更新都需用户明确确认

## 回显模板（对用户）

### 模板：流程路由确认（验证阶段）

```text
### 流程路由确认（验证阶段 | 请确认）
说明：
- 当前识别：这是"更新知识网络"请求
- 将进入流程：更新流程（FLOW_UPDATE）
- 你提供的信息摘要：{input_summary}
- 说明：这一步只确认流程方向，还不会开始更新
下一步：你可以回复"确认进入更新流程"，或先调整变更目标后再继续。
```

### 模板：更新前差异确认

```text
### 更新差异确认（更新流程 | 请确认）
说明：
- 更新目标：{kn_id_or_entity}
- 变更摘要：{diff_summary}
- 影响面：{affected_entities_and_fields}
下一步：请确认是否执行更新（回复：确认更新 / 调整变更）。
```

### 模板：更新结果回执

```text
### 更新执行结果（更新流程 | 结果）
说明：
- 成功项：{success_items}
- 失败项：{failed_items}
- 核验结果：{verify_summary}
下一步：如有失败项，请选择重试或调整后重试。若需详情可回复：展开更新详情 / 导出 YAML。
```
