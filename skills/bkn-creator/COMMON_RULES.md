# bkn-creator 通用门禁与约束

## 渐进式执行协议（统一适用于 CRUD）

所有流程必须遵循以下基准阶段序列，不得跳步：

1. `discover`：识别目标与上下文
2. `preview`：展示将执行内容（含影响面 / diff）
3. `confirm`：等待明确确认
4. `execute`：委托子技能执行
5. `verify`：核验结果完整性
6. `report`：向用户回执结果与下一步建议

> Never 并行跳阶段。任何写操作在 `confirm` 之前都不得执行。

各 `FLOW_*.md` 可在基准序列基础上插入领域专属子阶段（如提取流程在 `verify` 与 `report` 之间增设 `confirm_pending`），具体以对应文件为准。

## 明确确认判定规则

仅当用户回复命中以下确认语义，才视为通过门禁：

- 视角确认：`确认采用该视角` / `按这个视角继续` / `ok 这个视角`
- 清单确认：`确认清单` / `按此清单进入下一步` / `可以进入下一阶段`
- 写操作确认：`确认` / `确认更新` / `确认删除`

以下回复 Never 视为确认：

- 含糊反馈：`看一下` / `先这样` / `继续说` / `嗯`
- 提问或补充：带 `?` 的问题、或新增需求但未出现确认语义
- 纠错语句：`不对` / `改一下` / `先别继续`

## 流程路由确认门禁（验证阶段，适用于所有 CRUD）

在完成流程路由识别后，必须先完成“流程路由确认”再进入任一 `FLOW_*` 阶段。

- 流程路由确认通过语义：`确认` / `确认进入` / `按该流程继续` / `进入该流程`
- 流程路由确认未通过时：
  - Never 输出任何阶段化执行内容（如“阶段一”“Step 1”“根据 FLOW_*.md ...”）
  - Never 执行任何提取、生成、更新、删除、绑定、推送动作
  - 仅允许重复输出“流程路由识别结果 + 确认请求”
- 若已误入阶段内容，必须立即停止并回到流程路由确认请求

## 门禁强制执行（MUST）

| 门禁点 | 执行时机 | 拦截条件 | 违规处理 |
|--------|----------|----------|----------|
| 流程路由确认（验证阶段） | 任一 `FLOW_*` 前 | 未收到明确“进入对应流程”确认 | 仅允许输出“流程路由识别结果 + 确认请求”；禁止进入任一阶段内容 |
| A1/B1 视角确认 | 阶段一提取前 | 未收到明确视角确认 | 禁止任何对象类/关系类提取；若已误提取，立即中止并回到视角确认 |
| A2/B2/C1 清单确认 | 阶段二生成前 | 未收到明确清单确认 | 禁止生成 `.bkn` 文件 |
| 阶段二样式补齐检查 | 阶段三前 | `network.bkn` 未补齐默认 `icon/color` 或存在无颜色对象类 | 不得进入阶段三，必须先补齐样式 |
| 草案确认 | 阶段三执行前 | 未收到明确草案确认 | 禁止执行环境检查及后续写操作 |
| 阶段四执行与放行（统一） | 阶段四执行期 | 详见 `FLOW_CREATE.md` 阶段四 | 以 `FLOW_CREATE.md` 为唯一判定源，本表不重复定义 |
| 阶段五执行与回执（统一） | 阶段五执行期 | 详见 `FLOW_CREATE.md` 阶段五 | 以 `FLOW_CREATE.md` 为唯一判定源，本表不重复定义 |
| 网络更新确认 | 任何本地回填或线上更新前 | 未收到明确“确认更新网络/确认执行” | 禁止执行 `.bkn` 回填、`update`、`push`、`delete` 等更新动作 |
| 提取命名与分组校验 | 提取流程 `verify` 后、`report` 前 | 对象命名未归一化，或未区分 `explicit/inferred/pending` | 不得进入提取结果回显，必须先完成归一化与分组校验 |
| 待确认对象处理门禁 | 提取流程 `confirm_pending` 阶段 | `pending_objects` 非空但未完成处理决策 | 不得进入最终“清单确认”，必须先完成待确认对象处理 |

门禁确认语义补充：

- 用户必须回复“确认”或等价表达（如“是”“确定”“确认继续”）才可通过门禁
- “先这样”“继续说”“看一下”等模糊表达不视为确认
- 门禁未通过时，必须输出确认请求并等待用户响应，不得推进执行
- MUST：每轮对话至多发起一个门禁确认请求；Never 在同一轮输出中合并多个门禁（如视角确认 + 领域识别、清单确认 + 草案确认等）
- MUST：前一门禁未获用户确认前，不得提前执行或展示后续门禁所依赖的内容
- “跳过未绑定对象并继续”仅作用于未绑定对象；已绑定对象保持不变
- 阶段四细则（状态顺序、Step 4.5 必执行、放行条件、回填与确认）统一以 `FLOW_CREATE.md` 为唯一规则源，`COMMON_RULES.md` 不再重复定义
- 阶段五细则（同名冲突处理、推送确认、报告生成）统一以 `FLOW_CREATE.md` 为唯一规则源，`COMMON_RULES.md` 不再重复定义
- 新建网络默认样式固定为 `icon: icon-dip-graph`、`color: #0e5fc5`
- 对象类必须有 `color`，由 `bkn-creator` 在阶段二按随机策略补齐
- Never 自动更新网络；所有更新均需用户明确确认

## 术语词典（统一用词）

- 流程路由识别：指新增/提取/查找/更新/删除的流程判定
- 流程路由确认：指验证阶段的用户确认门禁
- 建模意图澄清：指新增流程阶段一（路径 A/B/C 的建模确认）
- 旧称“意图识别”“路由确认”“建模澄清”在对用户文案中不再使用

## 执行与委托边界

- MUST：凡需要执行 `kweaver` CLI，先读取 `../kweaver-core/SKILL.md`，再委托 `kweaver-core` 执行
- Never：`bkn-creator` 直接执行 CLI
- MUST：新增流程阶段二读取 `../create-bkn/SKILL.md` 生成 BKN 草案
- MUST：新增流程阶段一路径 C 读取 `../bkn-modeling-advisor/SKILL.md`，由其完成多轮建模对话
- MUST：新增流程除“输入模糊”外，出现“领域冲突未决 / `pending_objects >= 3` / 清单质量不足”时，也必须委托 `../bkn-modeling-advisor/SKILL.md`
- MUST：新增流程阶段四读取并委托 `../data-semantic/SKILL.md`，用于未绑定对象的数据视图语义匹配
- MUST：新增流程阶段四对“已绑定视图”也读取并委托 `../kweaver-core/SKILL.md`，使用 `kweaver dataview` 执行存在性校验
- MUST：新增流程中 A1/B1 视角确认通过前，Never 执行任何对象类/关系类提取动作
- MUST：新增流程默认主视角为“对象-属性-视图映射”；`实体-关系（ER）` 仅作为复杂关系场景的补充视角
- MUST：启用 ER 后仅用于关系骨架梳理，不得替代对象属性生成、属性映射完备性检查与可用性结论
- MUST：对象类/关系类提取在任何场景下先经过 `./references/DOMAIN_ROUTING.md` 统一调度
- MUST：领域提取仅使用 bkn-creator 内置 `./references/` 知识（含通用兜底），不进行外部委托
- MUST：对象类提取结果在回显前必须完成命名归一化；复合关系表达（如 `A -> B`）不得作为对象名
- MUST：对象类提取结果必须分组回显为 `explicit_objects`、`inferred_objects`、`pending_objects`，并对 `inferred_objects` 提供推断依据
- MUST：当 `pending_objects` 非空时，必须先发起“待确认对象处理”确认请求；未完成处理前不得进入“清单确认”
- MUST：关系类 `name` 使用中文业务名；英文仅可用于 `relation_id`（技术标识）

## 落盘与归档职责边界（MUST）

- MUST：`bkn-creator` 不负责目录分配、归档路径规则或归档回执格式
- MUST：凡涉及文件写入与归档（`.bkn`、报告、JSON 等），统一遵循并直接委托 skill `archive-protocol`
- MUST：`network_dir`、归档根目录、回读校验、`ARCHIVE_STATUS`/`ARCHIVE_ROOT`、`archive_grid` 等规则以 `archive-protocol` 为唯一规则源
- Never：在 `bkn-creator` 内重复定义 `archives/*` 路径模板或归档输出模板

## 全局硬约束

- MUST：对用户可见文案使用专业中性表达，Never 使用 emoji 或表情符号
- MUST：凡“对用户展示”的文本，统一使用“用户回显模板”并包含明确标识
- MUST：同一轮回显只允许一个主展示格式（文本摘要 / 表格 / YAML/JSON 三选一），禁止同一数据重复展示
- MUST：未获用户明确请求时，默认仅输出摘要，不主动输出 YAML/JSON 原始结构或表格明细
- MUST：同一轮对话中的用户可见内容保持稳定顺序（`标题（阶段 | 类型）` -> `正文区` -> `下一步`）
- MUST：优先复用各 `FLOW_*.md` 与 `references/DOMAIN_ROUTING.md` 中的“回显模板（对用户）”，Never 临时自拟结构
- MUST：用户可见文案禁止内部术语直出（如 `A1/B1/A2/B2/C1`、`门禁`）；统一改写为业务可读表达（如“建模视角确认”“确认步骤”）
- MUST：用户可见标题避免系统标签（如 `confirm |`、`execute |`），统一使用自然中文（如“请确认”“结果”）
- MUST：用户可见表格/清单禁止仅展示英文编码枚举（如 `keep/drop_candidate/core_business/view_derived/mapped/waived/blocked`）；必须使用中文标签（可附编码）
- MUST：数据定义字段（如 Description）仅允许稳定业务语义；映射不确定信息（如“待核实”“视图衍生”“候选字段”）必须放入独立映射备注字段，不得混入定义字段
- MUST：当输出“部分可用/不可用”结论时，必须同步给出判定依据与可执行改进项，不得仅给状态名
- MUST：ID 完整保存和使用，Never 截断
- MUST：所有创建/推送操作指定 `branch` 为 `main`（CLI 使用 `--branch main` 参数）
- MUST：输入不足时一次只追问最高优先级字段，不批量追问
- MUST：工具调用失败时停在当前阶段，向用户报错并等待指令，Never 自动重试或跳过
- Never：使用 `curl`、`kweaver-api.sh` 等已废弃方式
- Never：Windows 环境使用 `/tmp/` 路径
- Always：目录命名与落盘位置遵循 `archive-protocol` 规则

## 用户回显模板（统一格式）

凡需向用户展示的结果、确认请求、风险提示、执行回执，建议优先使用以下格式：

```text
### <本次输出标题>（<阶段> | <类型>）
说明：
- <要点 1>
- <要点 2>
下一步：<下一步动作或用户需确认事项>
```

补充约束：

- Never 在“用户回显模板”外混入关键结论，避免信息分散
- Always 将 `<阶段>` 与 `<类型>` 合并写入三级标题，Never 再单独输出“标题/阶段/类型”字段
- Always 默认使用“说明：”作为正文区标题；如需强调行动建议，可改为“建议：”，但同一条回显内不得混用
- 类型为“确认请求”时，`下一步` 必须包含可执行选项或明确确认短语
- Always 对用户优先展示中文业务词汇；若需要保留内部编码，统一采用“中文（编码）”格式
- 阶段四结果确认应优先复用 `FLOW_CREATE.md` 的“阶段四结果确认模板”，并显式包含 `step_4_5_summary` 与 `step_4_7_summary`
