# 理解状态机

## 状态码说明

| 状态码 | 名称 | 说明 | 是否可查询语义 |
|--------|------|------|----------------|
| 0 | 未理解 | 尚未进行语义理解 | ❌ |
| 1 | 理解中 | 语义理解进行中 | ❌ |
| 2 | 待确认 | 理解完成待确认 | ✅ |
| 3 | 已完成 | 已确认完成 | ✅ |
| 4 | 待确认(重) | 重新理解后待确认 | ✅ |
| 5 | 理解失败 | 语义理解失败 | ❌ |

## 状态流转图

```
┌─────────┐ generate   ┌─────────┐ status=2   ┌─────────┐
│ 0-未理解│ ───────────│ 1-理解中│ ───────────│ 2-待确认│
└─────────┘            └─────────┘            └────┬────┘
                                                   │
                              submit               │
                                                   ▼
┌─────────┐ generate   ┌─────────┐          ┌──────────┐
│ 3-已完成│ ◄──────────│4-待确认 │          │ 5-失败   │
└─────────┘            └─────────┘          └──────────┘
```

## 各状态处理逻辑

### 状态 0: 未理解

**处理**：直接触发理解

1. 调用 `POST /:id/generate`
2. 状态变为 1（理解中）
3. 开始轮询

### 状态 1: 理解中

**处理**：轮询等待

1. 每 10 秒调用 `GET /:id/status` 查询状态
2. 等待状态变为 2 或 3
3. 状态 = 2：调用 `POST /:id/submit` 提交确认
4. 状态 = 3：已完成

### 状态 2: 待确认

**处理**：提交 + 重新生成

1. 调用 `POST /:id/submit` 提交确认
2. 调用 `POST /:id/generate` 重新生成
3. 状态变为 1，进入轮询

### 状态 3: 已完成

**处理**：重新生成

1. 调用 `POST /:id/generate` 重新生成
2. 状态变为 1，进入轮询

### 状态 4: 待确认(已重新理解)

**处理**：同状态 3

1. 调用 `POST /:id/generate` 重新生成
2. 进入轮询

### 状态 5: 理解失败

**处理**：终止

1. 输出失败原因
2. 提示用户检查数据后重试

## API 调用时序

```
查询状态(0) → generate → 轮询(1) → 查询(2) → submit → generate → 轮询(1) → 查询(3) → 完成
```

## 批量处理

批量理解时，每个视图独立走状态机：

1. 并行查询所有视图状态
2. 根据状态分发处理
3. 并行轮询
4. 统一提交确认
5. 再次轮询直到全部完成
