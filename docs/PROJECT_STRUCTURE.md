# 项目目录结构说明

本文档用于说明当前仓库中各目录的职责，便于后续按统一规范组织代码、文档与发布内容。

## 顶层目录

### `.github`
- 用途：存放 GitHub 平台相关配置。
- 常见内容：
  - `workflows/`：CI/CD 流水线（如构建、测试、发布）。
  - `ISSUE_TEMPLATE/`：Issue 模板。
  - `PULL_REQUEST_TEMPLATE.md`：PR 模板。
- 建议：所有自动化流程与协作模板统一放在这里，避免散落在业务目录。

### `assets`
- 用途：存放项目静态资源。
- 当前子目录与文件示例：
  - `assets/logo/`：品牌与 README 用图（如 `light.png`、`dark.png`、**`kweaver-dip.png`**）。
  - `assets/kweaver-dip-architecture .png`：平台架构示意图（注意文件名中 `architecture` 与 `.png` 之间有空格，引用时需编码为 `%20`）。
- 建议：图片、图标等通用资源统一管理，避免在文档目录重复存放。

### `deploy`
- 用途：存放部署相关脚本与配置。
- 可放置内容：
  - 一键部署脚本（如 `deploy.sh`、PowerShell 脚本）。
  - 环境初始化脚本。
  - 部署说明文档（可在本目录下补充 `README.md`）。
- 建议：按环境区分（dev/test/prod）配置文件，减少误用风险。

### `docs`
- 用途：存放项目文档。
- 当前文件：
  - `docs/PROJECT_STRUCTURE.md`（本文件）。
  - `docs/qrcode.png`：README「交流社区」用二维码图。
- 可放置内容：
  - 架构说明、模块说明、使用手册、开发规范、FAQ。
- 建议：文档命名保持语义化，优先使用 Markdown。

### `release-notes`
- 用途：存放版本发布说明。
- 可放置内容：
  - 每个版本的变更摘要（新增、修复、优化、破坏性变更）。
  - 升级注意事项。
- 建议：按版本号命名，如 `v0.1.0.md`，便于追溯发布历史。

### `rules`
- 用途：存放项目规则、约束与规范定义。
- 可放置内容：
  - 代码规范、评审规则、提交规范、目录约定。
  - 业务规则或治理规则（按团队约定）。
- 建议：规则应可执行、可检查，避免仅口头约定。

### `sdk`
- 用途：统一存放各功能模块对外提供的**接口 SDK**（客户端库、API 封装、OpenAPI/Proto 生成代码等，具体形态按团队约定）。
- 可放置内容：
  - 面向前端、脚本或第三方集成的 SDK 源码与构建产物。
  - 各模块 SDK 的版本说明、变更记录与使用示例（可在本目录或 `docs/` 中引用）。
- 建议：
  - 与具体业务实现目录（如 `web`、`dsg`、`dip-studio` 等）分离，便于独立版本管理与发布。
  - 按模块或子包划分子目录（例如 `sdk/<module>/`），避免所有接口混在一个包内难以维护。

### `web`
- 用途：存放本项目所有前端代码。
- 可放置内容：
  - 通用前端应用代码、页面、组件与前端构建配置。
- 建议：前端公共能力（UI 组件、工具函数、请求封装）优先沉淀在该目录统一管理。

### `dsg`
- 用途：存放数据语义治理相关代码。
- 可放置内容：
  - 语义建模、治理规则、数据标准化与相关服务代码。
- 建议：按“模型定义 / 规则引擎 / 治理流程”拆分子目录，便于维护。

### `dip-studio`
- 用途：存放数字员工开发与管理相关代码。
- 可放置内容：
  - 数字员工的设计、配置、编排、发布与运维管理能力代码。
- 建议：建议在该目录下补充独立说明文档，明确开发流程与角色分工。

### `aistore`
- 用途：存放数字员工相关的应用表现层代码。
- 可放置内容：
  - 面向应用展示与交互的页面、模块和运行入口代码。
- 建议：重点关注应用展示一致性、可复用组件与发布规范。

### `chat-data`
- 用途：存放“数据分析员”数字员工相关代码。
- 可放置内容：
  - 数据分析场景下的数字员工逻辑、交互流程与能力实现代码。
- 建议：建议按“数据接入 / 分析能力 / 对话交互”组织子模块，便于迭代。

### `skills`
- 用途：存放 DIP 对外开发并交付的相关技能。
- 可放置内容：
  - 面向外部开发者或外部系统使用的技能包、技能定义与使用说明。
- 建议：每个技能建议包含版本信息、依赖说明、输入输出约定与示例，便于复用和发布。

## 推荐后续补充

为便于项目初始化，建议后续补充以下顶层文件：

- `README.md`：项目总览、快速开始、贡献方式。
- `README.zh.md`：中文说明（如需要中英文并行维护）。
- `.gitignore`：忽略构建产物、临时文件、环境文件等。
- `LICENSE`：开源许可证文本。
