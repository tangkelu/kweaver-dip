# 库表数据理解需求方案设计

> 基于业务对象管理界面的数据表结构设计（`t_data_view` 体系）
> 创建时间: 2026-01-30
> 更新时间: 2026-04-10（迁移为 `t_data_view` + 临时表/正式表分层）

## 一、设计概述

本设计方案实现了业务语义与数据存储的解耦，核心思想是：

- **业务对象（Business Object）**：业务实体的抽象
- **属性（Attribute）**：业务语义的抽象，关联数据视图字段语义补全结果
- **数据分层**：理解阶段统一写入临时表，提交阶段统一 Upsert 到正式表

## 三、数据表设计

> **【强制执行】本章节定义的所有数据表结构必须严格按照当前设计执行，不得擅自修改字段名称、类型、约束或索引。任何修改需经过完整评审流程。**

### 3.1 业务对象表 `t_business_object`

存储业务对象的基本信息，区分已发布业务对象和候选业务对象。

```sql
CREATE TABLE IF NOT EXISTS t_business_object (
    id             CHAR(36)     NOT NULL                       COMMENT '业务对象UUID（主键）',
    object_name    VARCHAR(100) NOT NULL                       COMMENT '业务对象名称',
    object_type    TINYINT      NOT NULL DEFAULT 0             COMMENT '对象类型：0-候选业务对象,1-已发布业务对象',
    form_view_id   CHAR(36)     NOT NULL                       COMMENT '关联数据视图UUID',
    status         TINYINT      NOT NULL DEFAULT 1             COMMENT '状态：0-禁用,1-启用',
    created_at     DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at     DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at     DATETIME(3)          DEFAULT NULL           COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='业务对象表';
```

### 3.2 业务对象属性表 `t_business_object_attributes`

存储业务对象的属性定义，关联到字段语义补全正式表 `t_data_view_field_semantic` 获取字段语义信息。

```sql
CREATE TABLE IF NOT EXISTS t_business_object_attributes (

    attr_name            VARCHAR(100) NOT NULL                       COMMENT '属性名称',
    created_at           DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at           DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at           DATETIME(3)          DEFAULT NULL           COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='业务对象属性表';
```

### 3.4 数据视图理解结果表 `t_data_view_understanding`

用于保存每个数据视图的理解结果（理解状态、视图业务名称、视图描述），并冗余保存视图技术信息（来源 `t_data_view`）。该表按 `view_id` 唯一，重复理解使用更新策略覆盖最新结果。

```sql
CREATE TABLE IF NOT EXISTS t_data_view_understanding (
    id                  CHAR(36)     NOT NULL COMMENT '记录UUID（主键）',
    view_id             VARCHAR(40)  NOT NULL COMMENT '关联 t_data_view.f_view_id',
    view_technical_name VARCHAR(255) NOT NULL DEFAULT '' COMMENT '视图技术名称（冗余，来源 t_data_view.f_technical_name）',
    view_group_id       VARCHAR(40)  NOT NULL DEFAULT '' COMMENT '视图分组ID（冗余，来源 t_data_view.f_group_id）',
    view_data_source_id VARCHAR(40)  NOT NULL DEFAULT '' COMMENT '数据源ID（冗余，来源 t_data_view.f_data_source_id）',
    view_type           VARCHAR(10)  NOT NULL DEFAULT '' COMMENT '视图类型（冗余，来源 t_data_view.f_type）',
    view_comment        VARCHAR(255) NOT NULL DEFAULT '' COMMENT '备注（冗余，来源 t_data_view.f_comment）',
    understand_status   TINYINT      NOT NULL DEFAULT 0 COMMENT '理解状态：0-未理解,1-理解中,2-待确认,3-已完成,4-已发布,5-理解失败',
    table_business_name VARCHAR(255) DEFAULT NULL COMMENT '视图业务名称',
    table_description   VARCHAR(300) DEFAULT NULL COMMENT '视图描述',
    created_at          DATETIME(3)  DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at          DATETIME(3)  DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at          DATETIME(3)  DEFAULT NULL COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (id),
    UNIQUE KEY uk_view_id (view_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='数据视图理解结果表';
```

**冗余字段来源规则（来自 `t_data_view`）**：


| `t_data_view_understanding` 字段 | 来源字段                           |
| ------------------------------ | ------------------------------ |
| `view_technical_name`          | `t_data_view.f_technical_name` |
| `view_group_id`                | `t_data_view.f_group_id`       |
| `view_data_source_id`          | `t_data_view.f_data_source_id` |
| `view_type`                    | `t_data_view.f_type`           |
| `view_comment`                 | `t_data_view.f_comment`        |


**更新策略**：

- 触发理解时，先按 `view_id` 读取 `t_data_view` 最新数据并写入（或更新）冗余字段。
- 提交时执行 Upsert：若 `view_id` 已存在则更新冗余字段与理解结果，不存在则新增。

**理解状态说明：**


| 状态值 | 状态名称 | 说明           |
| --- | ---- | ------------ |
| 0   | 未理解  | 尚未进行库表理解操作   |
| 1   | 理解中  | 正在进行库表理解操作   |
| 2   | 待确认  | 理解完成，待用户确认   |
| 3   | 已完成  | 已完成库表理解      |
| 4   | 已发布  | 业务对象已正式发布    |
| 5   | 理解失败 | AI理解失败，可重新尝试 |


### 3.5 字段语义补全正式表 `t_data_view_field_semantic`

用于保存提交确认后的字段语义结果，和临时表 `t_form_view_field_info_temp` 对应。

> **【重要】本表主键 `id` 与字段 UUID `form_view_field_id` 一致**（与临时表、业务对象属性表等关联），不再单独存储 `view_field_id` 列。  
> **【重要】本表包含从 t_data_view.f_fields JSON 解析后的冗余字段，查询时不再解析 JSON，提升性能。**

**JSON字段来源（t_data_view.f_fields）：**

```json
{
  "name": "id",                    // 字段技术名称
  "type": "string",                // 字段类型
  "comment": "",                   // 字段注释
  "display_name": "id",            // 显示名称（可作为业务名称）
  "original_name": "id",           // 原始字段名称
  "data_length": 255,              // 数据长度
  "data_accuracy": 0,              // 数据精度
  "status": "modify",              // 状态
  "is_nullable": "YES",            // 是否可空
  "business_timestamp": false,      // 是否业务时间戳
  "is_primary_key": { ... }       // 是否主键
}
```

```sql
CREATE TABLE IF NOT EXISTS t_data_view_field_semantic (
    id                      CHAR(36)     NOT NULL COMMENT '字段UUID（主键，与 form_view_field_id 一致）',
    view_id                 VARCHAR(40)  NOT NULL COMMENT '关联数据视图ID（对应 t_data_view.f_view_id）',

    -- 字段基础信息（冗余，从 t_data_view.f_fields JSON 解析，避免重复解析）
    field_tech_name         VARCHAR(255) DEFAULT NULL COMMENT '字段技术名称（对应 JSON.name）',
    field_type              VARCHAR(64)  DEFAULT NULL COMMENT '字段类型（对应 JSON.type）',
    field_comment          VARCHAR(500) DEFAULT NULL COMMENT '字段注释（对应 JSON.comment）',
    field_display_name     VARCHAR(255) DEFAULT NULL COMMENT '字段显示名称（对应 JSON.display_name）',
    field_original_name    VARCHAR(255) DEFAULT NULL COMMENT '原始字段名称（对应 JSON.original_name）',
    field_data_length      INT          DEFAULT NULL COMMENT '数据长度（对应 JSON.data_length）',
    field_data_accuracy    INT          DEFAULT NULL COMMENT '数据精度（对应 JSON.data_accuracy）',
    field_is_nullable      VARCHAR(10)  DEFAULT NULL COMMENT '是否可空（对应 JSON.is_nullable）',
    field_is_primary_key   TINYINT      DEFAULT NULL COMMENT '是否主键（对应 JSON.is_primary_key.Bool，1-是，0-否）',
    field_business_timestamp TINYINT    DEFAULT NULL COMMENT '是否业务时间戳（对应 JSON.business_timestamp）',

    -- 字段语义补全（AI 分析结果，提交后落库）
    field_business_name     VARCHAR(255) DEFAULT NULL COMMENT '字段业务名称（AI 分析）',
    field_role              TINYINT      DEFAULT NULL COMMENT '字段角色：1-业务主键, 2-关联标识, 3-业务状态, 4-时间字段, 5-业务指标, 6-业务特征, 7-审计字段, 8-技术字段',
    field_description       VARCHAR(300) DEFAULT NULL COMMENT '字段描述（AI 分析）',

    created_at              DATETIME(3)  DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at              DATETIME(3)  DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at              DATETIME(3)  DEFAULT NULL COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (id),
    KEY idx_view_id (view_id, deleted_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='字段语义补全正式表';
```

### 3.6 业务对象临时表 `t_business_object_temp`

用于在用户提交确认前临时保存业务对象的编辑操作，支持版本控制。

```sql
CREATE TABLE IF NOT EXISTS t_business_object_temp (
    id             CHAR(36)     NOT NULL                       COMMENT '业务对象UUID（主键）',
    form_view_id   CHAR(36)     NOT NULL                       COMMENT '关联数据视图UUID',
    in_use        TINYINT      NOT NULL DEFAULT 0            COMMENT '当前使用标识：0=历史版本, 1=当前使用',
    user_id        CHAR(36)                                         COMMENT '为空代表模型操作，不为空代表某用户操作',
    version        INT          NOT NULL DEFAULT 10            COMMENT '版本号（存储格式：10=1.0，11=1.1，每次递增1表示0.1版本）',
    object_name    VARCHAR(100) NOT NULL                       COMMENT '业务对象名称',
    created_at     DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at     DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at     DATETIME(3)          DEFAULT NULL           COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (id),
    KEY idx_form_view_version (form_view_id, version, deleted_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='业务对象临时表';
```

**数据来源：**

1. **AI 自动生成**：通过订阅 Kafka 消息 `data-understanding-responses`，接收 AI 服务分析完成后返回的业务对象数据
2. **用户手动编辑**：用户在页面上编辑业务对象名称时保存的数据

**使用说明：**

- 用户在页面编辑业务对象时，先保存到此临时表
- 点击提交确认后，将数据迁移到正式表 `t_business_object`
- `user_id` 为空时代表 AI 模型自动操作（通过 Kafka 订阅），不为空时代表具体用户操作
- `in_use` 用于标识当前使用的版本：`1=当前使用`，`0=历史版本`
  - AI 每次生成新识别结果时，会自动将新版本的 `in_use` 设置为 1，历史版本设置为 0
  - 删除识别结果时，只删除 `in_use = 1` 的数据
- AI 自动生成时递增版本号，初始版本为10（显示为1.0），用户手动编辑时版本号不变
- 支持重新识别，AI 重新识别时会创建新版本记录，旧版本记录保留作为历史

**使用说明：**

### 3.8 库表信息临时表 `t_form_view_info_temp`

临时记录库表的语义补全数据（包括 AI 自动生成和用户手动编辑），支持版本控制。

```sql
CREATE TABLE IF NOT EXISTS t_form_view_info_temp (
    id                   CHAR(36)     NOT NULL                       COMMENT '记录UUID（主键）',
    form_view_id         CHAR(36)     NOT NULL                       COMMENT '关联数据视图UUID',
    user_id              CHAR(36)                                         COMMENT '为空代表模型操作，不为空代表某用户操作',
    version              INT          NOT NULL DEFAULT 10            COMMENT '版本号（存储格式：10=1.0，11=1.1，每次递增1表示0.1版本）',

    -- 库表级别语义补全
    table_business_name  VARCHAR(255)        DEFAULT NULL            COMMENT '库表业务名称',
```

**处理逻辑**：

1. **查询当前理解状态**
  ```sql
   SELECT understand_status
   FROM t_data_view_understanding
   WHERE view_id = {view_id} AND deleted_at IS NULL;
  ```
2. **状态判断处理**
  **2.1 状态为 1-理解中**
   返回错误响应：
   **2.2 状态为 0-未理解、3-已完成、4-已发布**
   从正式表查询数据：
   **2.3 状态为 2-待确认**
   从临时表获取字段属性信息，字段基础信息从临时表冗余字段获取：
3. **查询参数对返回结果的影响（与接口协议一致的可观察行为）**
  在组装出字段列表后应用（实现可在内存或 SQL 中完成）：
  - `keyword`：不区分大小写；`field_tech_name` 或（非空的）`field_business_name` 包含该子串则保留。
  - `only_incomplete=true`：仅保留 `field_business_name` 为空 **或** `field_role` 为空的字段（与当前 `data_semantic` 实现一致）。

**数据来源说明**：

| 状态    | 库表信息                                    | 字段信息                       | 字段基本信息  |
| ----- | --------------------------------------- | -------------------------- | ------- |
| 0-未理解 | t_data_view + t_data_view_understanding | t_data_view_field_semantic | 正式表冗余字段 |
| 1-理解中 | 禁止调用                                    | -                          | -       |
| 2-待确认 | 临时表（优先）                                 | 临时表（优先）                    | 临时表冗余字段 |
| 3-已完成 | t_data_view + t_data_view_understanding | t_data_view_field_semantic | 正式表冗余字段 |
| 4-已发布 | t_data_view + t_data_view_understanding | t_data_view_field_semantic | 正式表冗余字段 |


- `table_tech_name`：来自 `t_data_view.f_technical_name`
- `table_business_name`、`table_description`：状态0/3/4来自 `t_data_view_understanding`，状态2临时表优先
- `field_tech_name`、`field_type`：状态0/3/4来自 `t_data_view_field_semantic` 冗余字段，状态2来自 `t_form_view_field_info_temp` 冗余字段
- `field_business_name`、`field_role`、`field_description`：状态0/3/4来自正式表，状态2临时表优先
- **本接口响应体不包含 `current_version`**；版本号仅在服务内部查询临时表时使用，不作为本接口字段返回。

**处理逻辑**：

1. **事务外校验**
   - **理解状态以 `t_data_view_understanding` 为权威落表**（§3.4）：按 **`view_id = Path 参数 id`** 且 **`deleted_at IS NULL`** 读取 **`understand_status`**。本方案下**不从已废弃的 `form_view` 表读取该字段**。
   - **仅 `understand_status = 2`（待确认）** 可提交，否则 `InvalidUnderstandStatus`。`0-未理解` / `1-理解中` / `3-已完成` / `4-已发布` / `5-理解失败` 均不允许提交。
   - 从业务对象临时表取 **`FindLatestVersionByFormViewId`**，入参为当前视图 id（与临时表 **`form_view_id`** 列存储的 `view_id` 一致），若版本为 `0` 则 `NoDataToCommit`。
   - 状态读取在 SQL 上的直接表达如下（与第三章一致）：
  ```sql
   SELECT understand_status
   FROM t_data_view_understanding
   WHERE view_id = {view_id} AND deleted_at IS NULL;
  ```

2. **数据库事务内执行（顺序与代码一致）**
   - 使用独立于 HTTP 请求的 context，**事务超时 120 秒**；任一步失败则整事务回滚。

   **2.1 业务对象（`processBusinessObjects`）**  
   - 按 **`form_view_id`（存值为 `view_id`）+ 上一步得到的 `latestVersion`** 读取 **`t_business_object_temp`**；读取 **`t_business_object`** 当前视图下全部记录，建立 **`object_name → 正式表 id`** 映射（§3.1）。  
   - 对临时表中每条记录：若正式表**已存在同名**业务对象，则**不重复插入**，仅按需更新与 **`t_data_view`** 主数据对齐所需的字段（以 §3.1 当前列集为准；历史代码若存在 `mdl_id` 等扩展列，以评审后 DDL 为准）；若**不存在同名**，则 **Insert** 正式表一行（主键 id 等与临时行一致，`object_type` / `status` 等为默认值，以仓库实现为准）。  
   - **当前实现不对**正式表中「临时当前版本已不再包含」的业务对象做逻辑删除；正式侧多余对象不在本接口中清理。

   **2.2 业务对象属性（`processBusinessObjectAttributes`）**  
   - 临时侧按 `form_view_id` 取**最新版本**的业务对象与属性；正式侧取当前视图下已有属性，按 `form_view_field_id` 建立「字段 → 属性」映射。  
   - 将临时属性通过临时业务对象 id → `object_name` → 正式业务对象 id 解析出 `business_object_id`；对每个临时属性：若该 `form_view_field_id` 在正式表**已有属性**则**更新**（业务对象 id、属性名等）；否则 **Insert** 新属性行。  
   - **当前实现不对**正式表中「临时最新版本已不再包含该字段属性」的行做逻辑删除。

   **2.3 库表业务信息（`updateFormViewInfo`）**  
   - 读取 **`t_form_view_info_temp`** 中当前 **`view_id`** 的**最新一条**；若无记录或查询失败则**跳过**。有则更新 **`t_data_view_understanding`** 中对应 **`view_id`** 行的 **`table_business_name`、`table_description`**（§3.4）。

   **2.4 字段业务信息（`updateFormViewFieldInfo`）**  
   - 读取 **`t_form_view_field_info_temp`** 当前 **`view_id`** 的**最新版本**列表；若无或为空则**跳过**。有则按第三章将字段级业务名称、角色、描述等 **写入/更新 `t_data_view_field_semantic`**（与临时表 **`form_view_field_id`** 对应主键 **`id`**，§3.5）。迁移期若代码仍写旧字段主表，以落库对齐第三章后的表名为准。

   **2.5 理解状态**  
   - 将 **`t_data_view_understanding.understand_status`** 更新为 **`3`（已完成）**（`view_id` = 路径 `id`）。

3. **事务结束与 HTTP 响应**
   - 事务成功提交后返回 HTTP 200，Body 见本节 **「响应体（成功 HTTP 200）」**（仅 `success: true`）。**`t_data_view_understanding.understand_status`** 已在步骤 2.5 更新为 **`3`（已完成）**，**不在**本响应 JSON 中返回。
   - 任一步失败则事务回滚，HTTP 非 200，错误体见 **§4.2**。

  - `t_form_view_info_temp` 和 `t_form_view_field_info_temp` 数据**保留**，不删除
  - 删除操作使用 `in_use = 1` 条件，确保只删除当前使用的识别结果
  - 历史版本数据（`in_use = 0`）保留，不参与当前业务流程
  - 删除后根据正式表是否有数据决定最终状态：
    - 有正式数据 → `3-已完成`

  - 如果需要重新生成，需要调用接口6重新生成数据

| 参数名 | 类型     | 必填  | 说明       |
| --- | ------ | --- | -------- |
| id  | string | 是   | 数据视图UUID |


**响应数据**：

```json
{
  "code": 200,                                 // int - 响应状态码
  "data": {
    "understand_status": 2,                   // int - 理解状态（0-未理解,1-理解中,2-待确认,3-已完成,4-已发布）
    "current_version": 10                     // int - 当前版本号（存储格式：10=1.0，显示时除以10）
  }
}
```

**处理逻辑**：

1. **查询理解状态**
  ```sql
   SELECT understand_status
   FROM t_data_view_understanding
   WHERE view_id = {view_id} AND deleted_at IS NULL;
  ```
2. **查询当前版本号**
  ```sql
   SELECT COALESCE(MAX(version), 10) AS current_version
   FROM t_form_view_info_temp
   WHERE form_view_id = {form_view_id} AND deleted_at IS NULL;
  ```

**说明**：

- 此接口用于前端轮询查询库表理解状态，特别是在 `1-理解中` 状态下
- 前端通过 `current_version` 字段判断数据是否有更新（版本号变化即表示数据更新）

#### 4.3.11 批量业务对象匹配

**接口路径**：`POST /api/data-semantic/v1/batch-object-match`

**接口说明**：批量匹配业务对象，输入为业务对象名称或视图ID列表。有视图ID时直接返回，有对象名字时优先从业务对象表模糊匹配，未匹配时从视图表模糊匹配并返回。

**请求参数**：


| 参数名                        | 类型     | 必填  | 说明                             |
| -------------------------- | ------ | --- | ------------------------------ |
| entries                    | array  | 是   | 业务对象列表，最多100条                  |
| entries[].name             | string | 是   | 业务对象名称（非空字符串）                  |
| entries[].data_source      | object | 否   | 给定的视图数据                        |
| entries[].data_source.id   | string | 否   | 视图ID（对应 t_data_view.f_view_id） |
| entries[].data_source.name | string | 否   | 视图名称                           |


**请求示例**：

```json
{
  "entries": [
    {
      "name": "客户信息",
      "data_source": {
        "id": "xxx",
        "name": "客户主档视图"
      }
    },
    {
      "name": "订单管理"
    }
  ]
}
```

**响应数据**：

```json
{
  "code": 200,
  "data": {
    "entries": [
      {
        "name": "客户信息",
        "data_source": [
          {
            "id": "mdl_xxx",
            "name": "客户主档视图",
            "object_name": "客户信息"
          }
        ]
      },
      {
        "name": "订单管理",
        "data_source": [
          {
            "id": "mdl_yyy",
            "name": "订单视图",
            "object_name": "订单管理"
          }
        ]
      }
    ],
    "need_understand": ["view_id_1", "view_id_2"]
  }
}
```

**响应字段说明**：


| 字段名                                 | 类型     | 说明                  |
| ----------------------------------- | ------ | ------------------- |
| entries                             | array  | 匹配结果列表              |
| entries[].name                      | string | 原始输入名称              |
| entries[].data_source               | array  | 匹配的视图列表             |
| entries[].data_source[].id          | string | 视图ID（mdl_id）        |
| entries[].data_source[].name        | string | 视图名称（TechnicalName） |
| entries[].data_source[].object_name | string | 业务对象名称              |
| need_understand                     | array  | 需要理解的视图ID列表（已去重）    |


**处理逻辑**：

1. **输入验证**：过滤空name的条目
2. **Step 1 - data_source有值**：如果给定了 `data_source.id`，直接追加到结果（返回id、name、object_name）
3. **Step 2 - 业务对象表模糊匹配**：
  ```sql
   SELECT id, object_name, form_view_id
   FROM t_business_object
   WHERE object_name LIKE '%{name}%' AND deleted_at IS NULL;
  ```
  - 找到匹配时，通过 `form_view_id` 查询视图信息获取 `mdl_id` 和 `TechnicalName`
4. **Step 3 - 视图表模糊匹配**：
  ```sql
   SELECT
     dv.f_view_id AS id,
     dvu.understand_status,
     dv.f_technical_name AS technical_name,
     dvu.table_business_name AS business_name
   FROM t_data_view dv
   LEFT JOIN t_data_view_understanding dvu ON dvu.view_id = dv.f_view_id AND dvu.deleted_at IS NULL
   WHERE dvu.table_business_name LIKE '%{name}%'
     AND dv.f_delete_time = 0;
  ```
5. **Step 4 - 状态检查**：
  - 无论什么状态都追加视图到 `data_source`
  - 仅当 `understand_status != 3` 时记录视图ID到 `need_understand`
  - 如果 `understand_status == 3`（已理解），查询业务对象表获取 `object_name`

**匹配优先级**：

1. 有 `data_source.id` → 直接返回
2. 业务对象表模糊匹配 → 返回匹配结果
3. 视图表模糊匹配 → 返回视图（未理解时记录到need_understand）

**错误响应**：


| HTTP状态码 | 错误码 | 说明                   |
| ------- | --- | -------------------- |
| 400     | -   | 请求参数错误（如entries为空数组） |

### 4.4 版本号管理规则

**版本号设计原则**：采用统一版本号管理，所有临时表共享同一个版本号。


| 接口               | 版本号变化 | 说明                                                                                                                                     |
| ---------------- | ----- | -------------------------------------------------------------------------------------------------------------------------------------- |
| 接口6 - 一键生成理解数据   | 递增    | 生成新版本时，递增所有临时表（`t_form_view_info_temp`、`t_form_view_field_info_temp`、`t_business_object_temp`、`t_business_object_attributes_temp`）的版本号 |
| 接口5 - 重新识别业务对象   | 递增    | 重新识别业务对象时，递增 `t_business_object_temp` 和 `t_business_object_attributes_temp` 的版本号                                                       |
| 接口7 - 保存库表信息补全数据 | 不递增   | 在当前最新版本上进行更新                                                                                                                           |
| 接口8 - 保存业务对象及属性  | 不递增   | 在当前最新版本上进行更新                                                                                                                           |
| 接口9 - 调整属性归属业务对象 | 不递增   | 在当前最新版本上进行更新                                                                                                                           |


**总结**：

- 接口6（一键生成）：递增所有临时表的版本号
- 接口5（重新识别）：只递增业务对象相关临时表的版本号
- 其他编辑操作：在当前最新版本上进行更新，不递增版本号

---

### 4.5 临时表数据处理策略

**临时表数据保留策略**：所有临时表数据在提交后**保留**，不删除。

**重新生成时的处理方式**：


| 场景             | 涉及临时表     | 处理方式             |
| -------------- | --------- | ---------------- |
| 接口6 - 一键生成理解数据 | 所有临时表     | 新增记录，版本号递增，旧记录保留 |
| 接口5 - 重新识别业务对象 | 业务对象相关临时表 | 新增记录，版本号递增，旧记录保留 |


**查询时数据来源**：

- 查询接口总是返回**最新版本**的数据
- 旧版本数据作为历史记录保留，不返回给前端

**说明**：

- 采用"新增记录"而非"更新记录"的方式，可以保留完整的修改历史
- 每次重新生成都会创建新版本的记录，便于追溯和对比
- 查询时通过 `in_use = 1` 筛选当前使用的版本，历史版本数据保留但不返回给前端

### 4.7 理解状态流转

> 理解状态统一存储于 `t_data_view_understanding.understand_status`，按 `view_id` 唯一维护。


| 当前状态   | 可调用接口 | 调用后状态                    |
| ------ | ----- | ------------------------ |
| 0-未理解  | 接口6   | 1-理解中（异步AI分析）            |
| 1-理解中  | -     | 2-待确认（AI成功后自动流转）         |
| 1-理解中  | -     | 5-理解失败（AI失败后自动流转）        |
| 2-待确认  | 接口3   | 3-已完成                    |
| 2-待确认  | 接口5   | 2-待确认（数据刷新）              |
| 2-待确认  | 接口4   | 0-未理解或3-已完成（取决于正式表是否有数据） |
| 3-已完成  | 接口4   | 0-未理解或3-已完成（取决于正式表是否有数据） |
| 3-已完成  | -     | 4-已发布（业务对象建模阶段）          |
| 4-已发布  | -     | -                        |
| 5-理解失败 | 接口6   | 1-理解中（重新尝试）              |


**说明**：

- `0-未理解` → `1-理解中`：用户点击"开始理解"按钮后立即设置
- `1-理解中` → `2-待确认`：AI异步处理成功后自动设置，前端需要轮询或接收通知来获取状态变化
- `1-理解中` → `5-理解失败`：AI异步处理失败后自动设置，用户可重新尝试
- `5-理解失败` → `1-理解中`：用户点击"开始理解"按钮后重新尝试
- **接口4删除回退逻辑**：删除业务对象临时数据后，检查正式表是否存在业务对象数据
  - 有正式数据 → 保持 `3-已完成`
  - 无正式数据 → 回退到 `0-未理解`

### 4.8 正式表增量更新策略

**设计背景**：保持原有业务流程不变，提交阶段统一将临时表数据同步到正式表。

**核心机制**：按业务唯一键执行 Upsert（存在即更新，不存在即新增）。

#### 4.8.1 Upsert 唯一键说明


| 正式表                            | 唯一键                                                |
| ------------------------------ | -------------------------------------------------- |
| `t_data_view_understanding`    | `view_id`                                          |
| `t_data_view_field_semantic`   | `id`（与 `form_view_field_id` 一致，按主键 Upsert）         |
| `t_business_object`            | `(form_view_id, object_name)`（结合业务约束）              |
| `t_business_object_attributes` | `(business_object_id, form_view_field_id)`（结合业务约束） |


#### 4.8.2 提交流程

1. 读取四类临时表当前版本数据：`t_form_view_info_temp`、`t_form_view_field_info_temp`、`t_business_object_temp`、`t_business_object_attributes_temp`
2. 依次同步到四类正式表：`t_data_view_understanding`、`t_data_view_field_semantic`、`t_business_object`、`t_business_object_attributes`
3. 对每条记录执行 Upsert：存在则更新，不存在则新增
4. 同步完成后更新 `t_data_view_understanding.understand_status = 3`

#### 4.8.3 示例场景

**场景：用户首次提交后再次编辑提交**


| 步骤  | 临时表状态        | 正式表操作  | 说明           |
| --- | ------------ | ------ | ------------ |
| 1   | 首次提交数据       | INSERT | 正式表无记录，新增    |
| 2   | 二次提交，部分数据有变化 | UPDATE | 命中唯一键，更新变化字段 |
| 3   | 二次提交，新增对象/属性 | INSERT | 未命中唯一键，新增    |


#### 4.8.4 优势总结


| 对比项  | 全量替换（旧方案） | Upsert 增量同步（新方案） |
| ---- | --------- | ---------------- |
| 业务流程 | 需额外删除/重建  | 保持原流程不变          |
| 变更范围 | 可能影响全量数据  | 仅更新变化项           |
| 可维护性 | 规则复杂      | 规则统一清晰           |
